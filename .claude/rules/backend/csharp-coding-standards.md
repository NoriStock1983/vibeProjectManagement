# C# コーディング規約

Microsoft 標準スタイル（.NET ガイドライン）に準拠する。以下のルールに従ってコードを書き、レビューする。

## フォーマット

- インデントはスペース4つを使用する。タブは使わない。
- 1行の長さは目安120文字以内に収める。超える場合は適切な位置で改行する。
- 波括弧 `{}` は常に新しい行に配置する（Allman スタイル）。
- `using` ディレクティブはファイルの先頭にまとめ、`System.*` を最初に、次にサードパーティ、最後にプロジェクト内の名前空間の順に並べる。

## 命名規則

- **クラス・構造体・インターフェース・列挙型・デリゲート**: PascalCase（例: `UserService`、`IRepository`）
- **メソッド・プロパティ・イベント・定数**: PascalCase（例: `GetUserById`、`IsActive`、`MaxRetryCount`）
- **フィールド（private）**: `_camelCase`（アンダースコアプレフィックス + camelCase、例: `_userName`）
- **ローカル変数・パラメータ**: camelCase（例: `userId`、`isValid`）
- **インターフェース**: 必ず `I` プレフィックスを付ける（例: `IUserRepository`、`ILogger`）
- **型パラメータ（ジェネリクス）**: 単一の場合は `T`、複数の場合は `TKey`・`TValue` のように意味のある名前を付ける。
- **非同期メソッド**: サフィックスに `Async` を付ける（例: `GetUserAsync`）
- **略語**: 2文字の略語は全大文字（例: `IO`）、3文字以上は PascalCase（例: `Http`、`Xml`）

## 型・変数

- 型が自明な場合は `var` を使用する。右辺の型が明確でない場合は明示的に書く。
- `string` / `int` などの組み込みエイリアスを使い、`String` / `Int32` は型メンバーアクセス時のみ使用する（例: `int.Parse`）。
- nullable 型（`T?`）を積極的に活用し、null 非許容の意図を型で表現する。プロジェクトで nullable reference types を有効にする（`<Nullable>enable</Nullable>`）。
- `const` は変更されない値に使う。実行時に決まる値は `static readonly` を使う。
- マジックナンバーは定数または列挙型に置き換える。

## クラス・構造体設計

- クラスの責務は単一にする（SRP）。
- 継承より合成を優先する。継承を使う場合は `sealed` を明示的に検討する。
- 公開 API には `virtual`・`abstract` を意図的に使い、継承を設計の一部として扱う。
- フィールドは `private` にし、必要に応じてプロパティで公開する。自動プロパティを積極的に使う。
- `record` 型はイミュータブルなデータ転送オブジェクト（DTO / Value Object）に使う。

## 例外処理

- 例外はフロー制御に使わない。想定内の失敗は戻り値（`bool`・`Result<T>` パターン等）で表現する。
- `catch` で例外を握りつぶさない。少なくともログに記録する。
- 例外をラップして再スローする場合は `throw;`（元のスタックトレースを保持）または `throw new MyException("...", ex);` を使う。`throw ex;` はスタックトレースをリセットするため使わない。
- `finally` または `using` ステートメントでリソースを確実に解放する。

## 非同期処理

- I/O バウンドな処理は `async/await` を使う。同期的にブロックする `.Result` や `.Wait()` は使わない（デッドロックの原因になる）。
- `async void` はイベントハンドラー以外では使わない。戻り値のない非同期メソッドは `Task` を返す。
- キャンセルが必要な処理には `CancellationToken` を引数で受け取り、適切に伝播させる。
- `ConfigureAwait(false)` はライブラリコードで使う。アプリケーションコードではコンテキストの要件に応じて判断する。

## LINQ

- LINQ はクエリ式よりメソッド構文を優先する（可読性の統一のため）。
- 複雑なクエリは変数に分割して命名し、意図を明確にする。
- `IEnumerable<T>` の遅延評価を意識し、複数回列挙する場合は `.ToList()` / `.ToArray()` で具体化する。

## コメント・ドキュメント

- `public` / `protected` なすべての型・メンバーには XML ドキュメントコメント（`///`）を書く。
- コメントは「何をしているか」ではなく「なぜそうしているか」を説明する。
- `TODO:` / `FIXME:` コメントには担当者または issue 番号を添える（例: `// TODO(n-satou): remove after migration #123`）。

## 名前空間・プロジェクト構成

- 名前空間はフォルダ構造と一致させる。
- `partial class` は自動生成コードとの結合にのみ使い、手書きコードの分割には使わない。
- `internal` を積極的に活用し、アセンブリ外に公開しないクラスを隠蔽する。

## テスト

- テストクラス名は `<対象クラス名>Tests`、テストメソッド名は `<対象メソッド>_<シナリオ>_<期待結果>` の形式にする（例: `GetUser_WhenNotFound_ReturnsNull`）。
- Arrange / Act / Assert の3セクションをコメントまたは空行で区切る。
- テスト間の依存関係を持たせない。各テストは独立して実行できる状態にする。
- モックは必要最小限にとどめ、統合テストと単体テストの役割を明確に分ける。

## ツール・静的解析

コードを書いたら以下を実行してから提出する：

```
dotnet build
dotnet test
```

可能であれば Roslyn アナライザー（`Microsoft.CodeAnalysis.NetAnalyzers`）および `dotnet format` も実行する。
