# Go コーディング規約

Go 標準スタイルに準拠する。以下のルールに従ってコードを書き、レビューする。

## フォーマット

- コードは必ず `gofmt`（または `goimports`）でフォーマットされた状態を維持する。手動での空白・インデント調整は行わない。
- インデントはタブ（`\t`）を使用する。スペースは使わない。
- 1行の長さに厳密な上限は設けないが、著しく長い行（目安120文字超）は読みやすく分割する。

## 命名規則

- **パッケージ名**: すべて小文字・単語区切りなし（例: `httputil`、`userservice`）。アンダースコア・ハイフン・大文字は使わない。
- **エクスポート識別子**: UpperCamelCase（例: `UserID`、`ParseToken`）
- **非エクスポート識別子**: lowerCamelCase（例: `maxRetry`、`parseToken`）
- **定数**: UpperCamelCase。`MAX_RETRY` のようなALL_CAPSは使わない。
- **インターフェース**: 単一メソッドのインターフェースはメソッド名 + `er` で命名する（例: `Reader`、`Stringer`）。
- **略語**: `URL`、`ID`、`HTTP` など略語は全大文字または全小文字で統一する（`userID` ○、`userId` ×）。
- **レシーバー名**: 型名の頭文字1〜2文字を使う（例: `func (u *User)`）。`self`、`this` は使わない。

## エラーハンドリング

- エラーは必ず呼び出し元で処理する。`_` で無視してよいのは、仕様上エラーが発生しないと確定している場合のみ。
- エラーメッセージは小文字で始め、句読点や改行で終わらない（`fmt.Errorf` でラップする際に先頭に追加されるため）。
- エラーのラップには `fmt.Errorf("context: %w", err)` を使い、呼び出しチェーンを辿れるようにする。
- カスタムエラー型が必要な場合は `errors.As` / `errors.Is` で判定できるよう `Unwrap()` を実装する。

## 関数・メソッド

- 関数は単一の責務を持つ。目安として、1関数が画面1スクロール（50行程度）を超える場合は分割を検討する。
- 戻り値が複数ある場合、エラーは最後に返す（Go 標準の慣例）。
- 名前付き戻り値は、ドキュメント目的に限定して使う。関数内で `return` のみ（naked return）を多用しない。

## コメント・ドキュメント

- エクスポートされるすべての識別子（型・関数・定数・変数）には godoc コメントを書く。コメントは識別子名で始める（例: `// UserID is ...`）。
- 非エクスポート識別子へのコメントは「なぜそうしているか」を説明する場合に書く。コードを読めば分かることは書かない。
- `TODO:` / `FIXME:` コメントには担当者または issue 番号を添える（例: `// TODO(n-satou): remove after migration #123`）。

## パッケージ構成

- `internal/` パッケージを活用し、外部に公開しない実装を隠蔽する。
- パッケージはひとつの明確な責務を持つ。「utility」「common」「misc」のような名前のパッケージは作らない。
- `init()` は設定ファイルの読み込みなど真に初期化が必要な場面に限定し、副作用が大きい処理は避ける。

## 型・変数

- 型の明示が不要な場合は `:=` による型推論を使う。ただし型が自明でない場合は明示的に書く。
- `interface{}` より `any` を使う（Go 1.18+）。
- ゼロ値を活用する。不要な初期化（`var s string = ""`）はしない。
- スライスの初期化は用途に応じて使い分ける。
  - 長さが分かっている場合: `make([]T, n)`
  - 追記専用: `var s []T`（nil スライスは append 可能）

## 並行処理

- goroutine を起動する際は必ずライフタイムを意識し、リークしないよう `context` やチャネルで終了を制御する。
- 共有状態へのアクセスには `sync.Mutex` または `sync/atomic` を使う。競合を生む設計は `go vet` / `go test -race` で検出する。
- チャネルはシグナル・所有権の移転に使い、単純な状態共有には Mutex を使う（"Don't communicate by sharing memory, share memory by communicating"）。

## テスト

- テストファイルは `_test.go` サフィックスを使い、テスト対象と同じパッケージ（内部テスト）または `_test` パッケージ（ブラックボックステスト）に置く。
- テスト関数名は `Test<対象関数名>_<シナリオ>` の形式にする（例: `TestParseToken_Expired`）。
- テーブル駆動テストを積極的に使う。同じ関数を複数パターンでテストする場合は `[]struct{ ... }` でケースを定義する。
- `t.Helper()` をヘルパー関数の先頭に呼び、エラー報告の行番号をテストコードに合わせる。

## ツール・静的解析

コードを書いたら以下を実行してから提出する：

```
go vet ./...
go test ./...
```

可能であれば `staticcheck` も実行する。
