# コード品質レビュー＆リファクタリングスキル（Good Code, Bad Code準拠）

「Good Code, Bad Code」（Tom Long著）の6つの品質特性でコードをレビューし、改善提案とリファクタリング済みコードを合わせて出力する。

## 6つの品質特性

| # | 特性 | 主なチェック観点 |
|---|------|----------------|
| 1 | **Correct（正確性）** | 仕様通りに動くか、エッジケース・境界値を処理しているか |
| 2 | **Readable（可読性）** | 命名が意図を伝えているか、ロジックの流れが追いやすいか |
| 3 | **Predictable（予測可能性）** | 驚きのある挙動がないか、副作用・グローバル状態の操作が明示されているか |
| 4 | **Robust（堅牢性）** | null/例外・異常値を適切に処理しているか、エラーが呼び出し元に伝わるか |
| 5 | **Testable（テスト容易性）** | 依存関係がDI可能か、純粋関数として切り出せるか、副作用が分離されているか |
| 6 | **Maintainable（保守性）** | 単一責任が守られているか、変更の影響範囲が小さいか、DRYか |

## 入力情報の収集

以下が揃っていない場合はまとめて確認する（一度の質問で済ませる）：

1. レビュー対象コード（ファイルパス or 貼り付け）
2. 言語・フレームワーク（自動判定できない場合）
3. 特に重視したい特性（省略可 — 省略時は6軸すべてチェック）
4. リファクタリング案のスコープ（省略可 — 「最小限の修正」か「設計レベルから見直す」か）

## 処理の流れ

1. コードを読み込み、全体像と責務を把握する
2. 6軸それぞれで **問題点・良い点** を特定し、優先度（High / Medium / Low）を付ける
3. High優先度の問題を中心に **リファクタリング済みコード** を生成する
4. 変更前後の差分と、なぜその変更が品質を上げるかの説明を添える
5. 結果を [assets/template.md](assets/template.md) の形式でファイルに保存する

## 出力形式

ファイル名：`code-review_[対象名]_YYYY-MM-DD.md`
保存先：`vibeProjectManagement/docs/code-reviews/`

## レビューの指針

- **問題の深刻度を明示する** — High（バグ・セキュリティリスク）/ Medium（品質低下）/ Low（改善余地）で分類する
- **良い点も必ず書く** — 問題点だけでなく、すでに良い書き方をしている箇所も明記する
- **リファクタリングは段階的に** — 一気に全部変えるのではなく、Highから順に適用できるよう提示する
- **言語の慣用句に合わせる** — Javaならオプショナル型、Pythonなら型ヒントなど、その言語のベストプラクティスに沿った改善を提案する
- 判断が難しい箇所は「トレードオフ」として両論を提示する

## 作成後の処理

ファイル保存後、ユーザーに保存パスを伝え、「リファクタリング案の適用や追加レビューがあればお知らせください」と案内する。
