# 前提条件・制約条件ログ作成スキル（PMBOK第8版準拠）

前提条件・制約条件ログ（Assumption Log）を作成し、Markdownファイルとして保存する。

## 目的

要件定義は前提条件に大きく依存する。前提を明文化しておくことで、後からの変更への対応が迅速になり、要件の再作業を最小限に抑えられる。PMBOK第8版では「不確実性への対応」がパフォーマンスドメインのひとつとして重要視されており、前提・制約の管理はリスク管理と一体として扱う。

## 入力情報の収集

以下の情報が揃っていない場合は、ユーザーにまとめて確認する（一度の質問で済ませる）：

**前提条件の例（確認ポイント）：**
1. 技術的前提（既存システムとの互換性・API仕様の安定性など）
2. ビジネス的前提（法規制の変更なし・組織体制の維持など）
3. リソース前提（必要なスキルセットを持つメンバーが確保できるなど）
4. ステークホルダー前提（要件収集への協力が得られるなど）
5. 外部依存（サードパーティサービスの継続・外部ベンダーの納期など）

**制約条件の例（確認ポイント）：**
1. スケジュール制約（リリース期限・法規制上の期日など）
2. 予算制約
3. 技術制約（使用技術・プラットフォームの制限）
4. 品質制約（性能要件・可用性・セキュリティ基準）
5. 法規制・コンプライアンス制約
6. リソース制約（チーム体制・利用可能な人員）

## 出力形式

ファイル名：`assumption-log_[プロジェクト名].md`
保存先：`vibeProjectManagement/docs/requirements/`

テンプレートは [assets/template.md](assets/template.md) に定義されている。必ずこのファイルを読み込み、その構造に従って出力する。

## 作成の指針

- **前提が崩れたときの影響を必ず記載する** — 「この前提が成立しなかった場合、どうなるか」を各前提に記載する
- **制約はリスクと紐付ける** — 制約が守れなかった場合のリスクを明示する
- **定期的に見直しが必要** — 前提は変化するため、見直し頻度・責任者を定める
- 既存の文書（プロジェクト憲章・ビジネスケース）に記載された前提・制約を参照・統合する

## 作成後の処理

ファイル保存後、ユーザーに保存パスを伝え、「内容の確認・修正があればお知らせください」と案内する。
