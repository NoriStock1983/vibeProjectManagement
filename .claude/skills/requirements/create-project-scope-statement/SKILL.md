# プロジェクトスコープ記述書作成スキル（PMBOK第8版準拠）

プロジェクトスコープ記述書（Project Scope Statement）を作成し、Markdownファイルとして保存する。

## 目的

スコープの境界を明確にすることで、要件の際限ない追加（スコープクリープ）を防ぐ。何がスコープ内で何がスコープ外かを関係者全員が理解するために必要。

## 入力情報の収集

以下の情報が揃っていない場合は、ユーザーにまとめて確認する（一度の質問で済ませる）：

1. プロジェクトスコープの詳細な説明（何を実施するか）
2. 成果物一覧（deliverables：何を作るか）
3. スコープ外の明示（何をやらないか・将来フェーズに回すものを含む）
4. 完了の定義（Definition of Done：何をもってプロジェクト完了とするか）
5. 受け入れ基準（成果物が承認される条件）
6. 成果物の承認者

要件文書（requirements-documentation）・プロジェクト憲章が既に作成されている場合は参照する。

## 出力形式

ファイル名：`project-scope-statement_[プロジェクト名].md`
保存先：`vibeProjectManagement/docs/requirements/`

テンプレートは [assets/template.md](assets/template.md) に定義されている。必ずこのファイルを読み込み、その構造に従って出力する。

## 作成の指針

- **スコープ外を明示的に書く** — 「スコープ外」セクションは「スコープ内」と同様に重要。曖昧な要件が後で追加されないよう、境界を明確にする
- **完了の定義を具体的に** — 「システムが動く」ではなく「本番環境での受け入れテスト合格・利用者への操作説明完了・運用引き継ぎ完了」のように具体化する
- **受け入れ基準は検証可能に** — 承認者が確認できる形で記述する
- スコープクリープ防止のため、このドキュメントをベースラインとして管理する

## 作成後の処理

ファイル保存後、ユーザーに保存パスを伝え、「内容の確認・修正があればお知らせください」と案内する。
