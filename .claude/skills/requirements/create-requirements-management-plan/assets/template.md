## 3. 要件の分析・優先付け

### 3.1 優先付け手法

**使用手法：MoSCoW法**

| 分類 | 説明 | 判断基準 |
|------|------|---------|
| Must Have（必須） | プロジェクト成功に不可欠な要件 | これがなければリリース不可 |
| Should Have（重要） | 重要だが代替手段がある要件 | 次フェーズでも許容できる |
| Could Have（あれば良い） | あれば便益はあるが、なくても問題ない要件 | リソースに余裕があれば対応 |
| Won't Have（今回はしない） | 今回のスコープ外と合意した要件 | 将来のフェーズ候補 |

### 3.2 分析プロセス

1. 要件を収集し、重複・矛盾がないか確認する
2. ビジネス目標への貢献度・実装コスト・リスクを評価する
3. ステークホルダーと協議して優先度を決定する
4. プロジェクトマネージャーが最終的な優先度を確定する

## 6. 変更管理プロセス

### 6.1 変更申請

変更を希望する者は、以下の情報を記載した「要件変更申請書」を提出する：

- 変更対象の要件ID・要件名
- 変更内容（現在 → 変更後）
- 変更理由・背景
- 変更による影響範囲（スコープ・スケジュール・コスト・品質）

### 6.2 変更の評価・承認フロー

```
変更申請
    ↓
PM が影響評価を実施（スコープ・コスト・スケジュール）
    ↓
影響度による承認権限の振り分け
    ↓
[影響小] PM単独承認 → 実施
[影響中] PM + スポンサー承認 → 実施
[影響大] 変更管理委員会（CCB）承認 → 実施 / 却下
    ↓
承認済み変更を要件文書・RTMに反映
```

### 6.3 変更管理委員会（CCB）

| 役割 | 氏名 |
|------|------|
| 委員長 | |
| 委員（PM） | |
| 委員（スポンサー代表） | |

### 6.4 緊急変更の取り扱い

緊急度が高く通常プロセスを経られない場合は、PMとスポンサーが口頭で暫定承認し、事後に正式な変更申請書を提出する。

## 7. トレーサビリティの管理

### 7.1 管理ツール

- **管理ツール**：要件トレーサビリティマトリクス（RTM）
- **保管場所**：`vibeProjectManagement/docs/requirements/`
- **更新頻度**：要件変更・設計変更・テスト実施の都度更新

### 7.2 トレーサビリティの対象

各要件は以下の項目と紐付けて管理する：

- ビジネス目標（Business Case）
- 設計成果物（設計書・画面仕様書）
- 実装（ソースコード・機能モジュール）
- テストケース（テスト計画・テスト結果）
