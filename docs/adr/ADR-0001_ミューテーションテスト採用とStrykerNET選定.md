# ADR-0001: ミューテーションテスト採用とStryker.NET選定

## ステータス

採用

## 背景

新規C#プロジェクトにおいて、ユニットテストのコードはClaude Codeが自動生成する方針を採用した。この場合、カバレッジ計測（Coverlet）のみでは「コードが実行された」という事実しか担保できず、テスト自体が意味のある検証を行っているかどうかを保証できない。

AIが生成したテストは「実行は通るが何も検証していない」テストを含むリスクがあるため、テストの品質を客観的に評価する手段としてミューテーションテストの導入を検討した。

## 検討した選択肢

### 選択肢 A: ミューテーションテストを導入しない（カバレッジのみ）

- **概要**：Coverletによるカバレッジ計測のみでテスト品質を管理する
- **メリット**：追加ツール不要、CI/CDへの影響が少ない
- **デメリット**：カバレッジ100%でもテストが実際にバグを検出できるかは保証できない。特にAI生成テストでは「通るだけのテスト」が混入するリスクが高く、品質担保として不十分

### 選択肢 B: VisualMutator

- **概要**：Visual Studio統合のGUIベースなミューテーションテストツール
- **メリット**：Visual Studio上でGUI操作が可能、セットアップが簡単
- **デメリット**：開発がほぼ停止しており、.NET Core / .NET 5以降に非対応。実用レベルでの採用は不可

### 選択肢 C: Stryker.NET

- **概要**：.NET向けのオープンソースミューテーションテストフレームワーク
- **メリット**：.NET 8/9を完全サポート、HTMLレポートによる視覚的な確認が可能、CI/CDへの組み込みが容易、しきい値設定によるビルド失敗制御が可能、インクリメンタルモードで差分のみ実行できる
- **デメリット**：大規模コードベースでは実行時間が長くなる、設定ファイル（stryker-config.json）の初期調整が必要

## 決定

**Stryker.NETを採用する。**

C#のミューテーションテストツールとして現在も活発に開発が継続されている実質唯一の選択肢であり、モダンな.NET環境への対応・CI/CD統合・視覚的なレポートの点で十分な実用性を持つ。

また、テストコードをClaude Codeが自動生成する本プロジェクトの方針上、AI生成テストの品質を客観的に検証する手段としてミューテーションテストの重要度は通常より高い。カバレッジだけではAI生成テストの妥当性を保証できないため、Stryker.NETによるミューテーションスコアを品質指標として併用する。

## 影響

### ポジティブな影響

- AI生成テストの品質を客観的に評価できる
- 「実行されたコード」だけでなく「バグを検出できるか」を継続的に確認できる
- HTMLレポートにより、テストが不十分な箇所を視覚的に特定できる

### ネガティブな影響・トレードオフ

- CI/CDパイプラインの実行時間が増加する
- 大規模化に伴い、ミューテーションテストの実行時間が問題になる可能性がある

### 対応が必要なタスク

- [ ] Stryker.NETのインストールと初期設定（stryker-config.json作成）
- [ ] CI/CDへの組み込みと実行タイミングの決定（毎PR or 夜間バッチ等）
- [ ] ミューテーションスコアのしきい値設定（現実的な初期値の検討）
