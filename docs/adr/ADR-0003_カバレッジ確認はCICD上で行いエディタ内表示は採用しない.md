# ADR-0003: カバレッジ確認はCI/CD上で行い、エディタ内表示は採用しない

## ステータス

採用

## 背景

カバレッジ計測にCoverlet + ReportGeneratorを採用する方針のもと、カバレッジの確認方法として「VSCode拡張機能（Coverage Gutters）によるエディタ内リアルタイム表示」と「CI/CDパイプラインでの自動計測・GitHub上でのレポート確認」の2つのアプローチを検討した。

本プロジェクトではテストコードをClaude Codeが自動生成する方針（ADR-0001参照）のため、開発者がテストを書きながらリアルタイムにカバレッジを確認するワークフローは発生しにくい。この前提のもと、Coverage Guttersの必要性を評価した。

## 検討した選択肢

### 選択肢 A: Coverage Gutters（VSCode拡張）を導入する

- **概要**：lcov形式のカバレッジデータを読み込み、VSCodeのエディタ上でカバレッジを行単位でリアルタイム表示する
- **メリット**：ローカルでのフィードバックループが短い、テストを書きながら即時確認できる
- **デメリット**：テストをClaude Codeが自動生成する本プロジェクトでは「書きながらリアルタイム確認」のユースケースが発生しにくく、恩恵が小さい。開発者ごとのセットアップ手順が増える

### 選択肢 B: CI/CD + ReportGeneratorによるGitHub上での確認

- **概要**：PRまたはpush時にCoverletでカバレッジを収集し、ReportGeneratorでHTMLレポートを生成してCI/CD上で確認する
- **メリット**：チーム全員が同一条件でカバレッジを確認できる。個人環境のセットアップ不要。Stryker.NETのHTMLレポートと確認フローを統一できる
- **デメリット**：ローカルでの即時確認はできない（CI/CDの完了を待つ必要がある）

## 決定

**CI/CD + ReportGeneratorによるGitHub上でのカバレッジ確認を採用し、Coverage Guttersは導入しない。**

テストコードをClaude Codeが自動生成する本プロジェクトでは、開発者がテストを書きながらリアルタイムにカバレッジを確認するワークフローが存在しないため、Coverage Guttersの主要なメリットを享受できない。

カバレッジの詳細確認はReportGeneratorのHTMLレポート、テスト品質の評価はStryker.NETのHTMLレポートで代替できるため、エディタ内表示は不要と判断した。

なお、Coverage Guttersは必須ツールとしては位置づけないが、個人の好みで導入することは妨げない。

## 影響

### ポジティブな影響

- 開発者ごとのローカル環境セットアップ手順が削減される
- カバレッジ確認フローがCI/CDに統一され、チーム全員が同一条件で評価できる
- Stryker.NETのレポートとカバレッジレポートの確認フローを統一できる

### ネガティブな影響・トレードオフ

- ローカルでのカバレッジ即時確認ができない（CI/CDの完了待ちが必要）

### 対応が必要なタスク

- [ ] CI/CDパイプラインにCoverlet + ReportGeneratorの実行ステップを組み込む
- [ ] ReportGeneratorの出力をGitHub Actions等で参照可能な形式で公開する設定を行う
