# ADR-0004: バックエンド自動テスト戦略

## ステータス

採用

## 背景

C#バックエンドに対して自動テストを導入するにあたり、以下を検討課題とした。

- 実施するテストの種類と、実施しないテストの明確化
- 各テストの実行タイミング（コミット時・PRマージ前・定期実行）
- バックエンドとして最も守るべき品質特性の定義

検討の結果、**バックエンドが最優先で守るべき品質特性は「正確性」** と定義した。
正確性とは、フロントエンドからの処理依頼を正確に処理し、期待通りの結果を返すことを指す。
具体的には以下の5層で構成される。

| レイヤー | 意味 |
|---|---|
| 計算・業務ロジック | 処理の演算結果が正しいか |
| データ変換 | リクエストの受け取りと変換が正確か |
| データ永続化 | DBへの書き込み・読み出しが正確か |
| APIレスポンス | フロントエンドが期待する形式・内容で返せるか |
| エラーハンドリング | 異常系を正確に処理・返却できるか |

この正確性の定義を軸に、各テスト種別の採否とタイミングを決定した。

## 検討した選択肢

### 選択肢 A: 全テストを毎回実行する

- **概要**：静的解析・Unit TEST・Integration TEST・E2E TEST・ミューテーションテストを常時実行
- **メリット**：品質担保が最大化される
- **デメリット**：実行時間が現実的でなく、開発フローが停滞する。特にE2EテストとミューテーションテストはCI実行時間が大幅に増加する

### 選択肢 B: 実行タイミングを分けてテスト種別ごとに最適化する（採用）

- **概要**：テストの目的・速度・コストに応じて実行タイミングを段階的に分ける
- **メリット**：フィードバックループを保ちながら正確性を担保できる。CIの実行時間を現実的な範囲に抑えられる
- **デメリット**：タイミング設計が複雑になる。設計の意図が失われると形骸化するリスクがある

### 選択肢 C: Unit TESTのみ実施する

- **概要**：軽量なUnit TESTのみでコストを最小化する
- **メリット**：実行時間が短い
- **デメリット**：DBや外部サービスとの連携時の正確性を保証できない。「正確性ファースト」の方針と整合しない

## 決定

**選択肢 B を採用する。** 実行タイミングを3段階に分け、テスト種別ごとに最適化する。

### 実行タイミングと対象テスト

#### Stage 1 — 毎コミット・PRプッシュ時（ローカル含む）

| テスト種別 | 実行場所 | 理由 |
|---|---|---|
| 静的解析 | pre-commit フック | 秒単位で完了し、コミット前にコード品質の一次チェックが行える |
| Unit TEST | CI/CD（PR作成・更新時） | 業務ロジックの正確性を最速でフィードバックできる |

> **補足：** pre-commitフックには静的解析のみを置く。Unit TEST以上の重いテストをpre-commitに含めると、コミットのたびに開発体験が著しく低下するため除外する。

#### Stage 2 — PRマージ前（必須ゲート）

| テスト種別 | 実行場所 | 理由 |
|---|---|---|
| Integration TEST | CI/CD + branch protection | DBやAPIとの実際のデータフローを通じた正確性を保証する。mainブランチへのマージ前に必ず通過させることでブランチ破壊を防ぐ |
| APIコントラクトテスト | CI/CD + branch protection | フロントエンドが期待するAPIの入出力仕様との一致を検証する。E2Eテストより高速かつ独立して実行できる |

> **補足：** Integration TESTはマージ「前」にCI/CDで実行し、branch protectionによって必須チェックとして設定する。マージ後に実行した場合、mainブランチが壊れるリスクがあるため採用しない。

#### Stage 3 — 定期実行（週次 or スプリント単位）

| テスト種別 | 実行場所 | 理由 |
|---|---|---|
| ミューテーションテスト | CI/CD スケジュール実行 | 業務ロジックの核心部分に絞って実行する。テスト自体の品質（正確性を保証するテストが本当にバグを検出できるか）を定期的に評価する（ADR-0001参照） |

> **補足：** 対象範囲は全コードではなく、業務ロジックの中核部分に限定する。全コードに適用すると定期実行でも現実的な時間に収まらない可能性があるため、範囲は別途設定ファイルで管理する。

### 採用しないテストと理由

| テスト種別 | 判断 | 理由 |
|---|---|---|
| バックエンドのE2Eテスト | **採用しない** | フロントエンドが「ユーザ体験」目的でE2Eテストを担当する。バックエンドが守るべき「APIレスポンスの正確性」はAPIコントラクトテストで代替できるため、重複投資を避ける |

### APIコントラクトテストについて

APIコントラクトテストはPactを採用する。

- フロントエンド（Consumer）がPact Consumer Testを記述し、契約書（pactファイル）を生成する
- バックエンド（C# / Provider）がPactNetを使用して契約書を読み込み、実際のエンドポイントが仕様を満たすかを検証する
- 契約書はPact Brokerで管理し、フロントエンド・バックエンド双方が独立してテストを実行できる構成とする

## 影響

### ポジティブな影響

- 「正確性」という目標から逆算したテスト設計となり、テストの意図が明確になる
- PRマージ前にIntegration TESTとAPIコントラクトテストが必須ゲートとなるため、mainブランチの正確性を継続的に保証できる
- バックエンドのE2Eテストを廃止することで、CI実行時間を削減しつつ正確性の保証は維持できる
- フロントエンドとバックエンドがAPIコントラクトを共有することで、インターフェースの不一致を早期に発見できる

### ネガティブな影響・トレードオフ

- APIコントラクトテストの導入にはフロントエンドとの協調が必要であり、初期セットアップコストが発生する
- Integration TESTをPRマージ前の必須ゲートにすることで、CIの実行時間がStage1より長くなりPRのフィードバックループが延びる
- ミューテーションテストの対象範囲をコード上でどう定義・管理するかの設計が別途必要

### 対応が必要なタスク

- [ ] CI/CDパイプラインの構築（GitHub Actions 等）
- [ ] branch protectionルールの設定（Stage2テストを必須ゲートに指定）
- [ ] pre-commitフックの設定（静的解析のみ）
- [ ] PactNetの導入とProvider Test実装
- [ ] Pact Brokerのセットアップ（OSS版 or PactFlow）
- [ ] ミューテーションテストの対象範囲定義（stryker-config.json の scope 設定）
