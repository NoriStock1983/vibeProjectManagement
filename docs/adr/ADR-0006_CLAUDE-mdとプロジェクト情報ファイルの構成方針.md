# ADR-0006: CLAUDE.md とプロジェクト情報ファイルの構成方針

## ステータス

採用

## 背景

Claude Code を活用して、仕様書・議事録・WBS・週次報告などの定型的かつ価値を生みにくい作業を Claude に委譲することを想定している。これらの作業を適切に遂行するには、Claude がプロジェクトの目的・ステークスホルダー・スケジュール・アーキテクチャなどの文脈情報を参照できる必要がある。

一方で、これらの情報は人間同士のコミュニケーションにおいても参照される重要なドキュメントであり、Claude 専用の設定ファイルと同じ場所に混在させることは適切ではない。

また、CLAUDE.md に全情報を直接記載すると、更新頻度の異なる情報が1ファイルに集中し、保守性が低下する。単一責任の原則に基づき、情報の性質ごとにファイルを分離する構成が必要と判断した。

## 検討した選択肢

### 選択肢 A: CLAUDE.md にすべての情報を直接記載する

- **概要**：プロジェクト概要・ステークスホルダー・スケジュール・アーキテクチャをすべて CLAUDE.md に書く
- **メリット**：ファイル数が少なくシンプル。Claude が確実に参照できる
- **デメリット**：更新頻度の異なる情報（スケジュールは高頻度、概要は低頻度）が混在し保守性が低い。CLAUDE.md が肥大化する。単一責任の原則に違反する

### 選択肢 B: `.claude/context/` に分割する

- **概要**：Claude が参照するプロジェクト情報を `.claude/context/` 配下に独立ファイルとして分割し、CLAUDE.md から参照させる
- **メリット**：情報ごとにファイルが分かれ保守性が高い。`.claude/` に集約されるため Claude の設定と一元管理できる
- **デメリット**：`.claude/` は Claude 専用の設定領域であり、人間が日常的に参照するドキュメントを置く場所として直感的でない。人間同士のコミュニケーションで参照しにくい

### 選択肢 C: `docs/project/` に分割し、CLAUDE.md から参照する

- **概要**：プロジェクト情報を `docs/project/` 配下に分割する。`.claude/` は Claude 専用設定のみとし、CLAUDE.md から `docs/project/` への参照を明記する
- **メリット**：単一責任の原則に沿った構成。人間が参照しやすい場所にドキュメントが置かれる。Claude も CLAUDE.md の指示を通じて参照できる。`.claude/` の役割が明確になる
- **デメリット**：CLAUDE.md と `docs/` の2箇所を意識する必要がある

## 決定

**選択肢 C を採用する。**

プロジェクト情報（概要・ステークスホルダー・スケジュール・アーキテクチャ）は `docs/project/` 配下に配置し、情報の種類ごとにファイルを分割する。`.claude/` は Claude の動作設定（rules・skills・agents）のみを管理する領域とする。CLAUDE.md はエントリポイントとして機能し、Claude が参照すべきファイルへのパスと作業指示を記載する。

これにより、更新頻度の異なる情報を独立して管理でき、かつ人間・Claude の両者が自然な形でアクセスできる構成となる。

## ファイル構成

```
vibeProjectManagement/
├── CLAUDE.md                        # エントリポイント + docs/ への参照 + Claude への指示
├── docs/
│   ├── project/
│   │   ├── overview.md              # プロジェクト概要・目的・フェーズ
│   │   ├── stakeholders.md          # ステークスホルダー一覧・役割
│   │   ├── schedule.md              # スケジュール・マイルストーン
│   │   └── architecture.md          # アーキテクチャ情報
│   └── adr/                         # Architecture Decision Records
└── .claude/
    ├── rules/                       # Claude のコーディング規約
    ├── skills/                      # Claude のスキル定義
    └── agents/                      # Claude のエージェント定義
```

## 影響

### ポジティブな影響

- 情報の性質（更新頻度・参照者）ごとにファイルが分離され、保守性が向上する
- `.claude/` の役割が「Claude 専用設定」に限定され、構造が明確になる
- `docs/project/` は人間同士のコミュニケーションにも活用できる
- 単一責任の原則に沿った構成により、将来の拡張がしやすくなる

### ネガティブな影響・トレードオフ

- CLAUDE.md と `docs/` の2箇所を意識して管理する必要がある
- Claude が `docs/project/` を参照するには CLAUDE.md に明示的な参照指示を記載する必要があり、記載漏れが起きると Claude が文脈を把握できなくなる

### 対応が必要なタスク

- [ ] `docs/project/overview.md` の作成
- [ ] `docs/project/stakeholders.md` の作成
- [ ] `docs/project/schedule.md` の作成
- [ ] `docs/project/architecture.md` の作成
- [ ] `CLAUDE.md` に参照指示と作業ルールを記載
